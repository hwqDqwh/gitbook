MySQL45学习笔记

基础架构

整体架构

大体来说，MySQL 可以分为 Server 层和存储引擎层两部分。 Server 层包括连接器、查询缓存、分析器、优化器、执行器等，以及所有的内置函数（如日期、时间、数学和加密函数等），所有跨存储引擎的功能都在这一层实现，比如存储过程、触发器、视图等。 存储引擎层负责数据的存储和提取。InnoDB 从 MySQL 5.5.5 版本开始成为默认存储引擎。

连接器

连接器负责跟客户端建立连接、获取权限、维持和管理连接。 要执行 MySQL 操作首先需要建立 MySQL 连接，例如使用工具或命令行 mysql -h -u -P 等。连接发起后首先会完成经典的 TCP 握手，然后连接器开始认证身份，用的就是输入的用户名和密码。 认证身份顺序是：

  1. 账密检查
  2. 权限检查，主要检查用户(来源IP)是否有链接某库的权限等

注意：连接建立后整个链接中执行的操作权限均依赖【权限检查】时读取到的用户权限，意味着建立连接后如果修改了权限需要重新建立 MySQL 链接。

链接自动断开

链接建立后如果长时间无操作，连接器会自动断开，这个时间由配置 wait_timeout 控制，默认 8 小时。

长链接和短链接

MySQL 中区分长链接和短链接，顾名思义，长链接指建立连接后一直维持链接不断开，多次操作均使用相同链接。短链接则指每次只执行少数几个操作即断开，下次需要再重新建立连接。

优缺点： 长链接的优点是减少建立连接的资源消耗（扩展，建立连接过程中产生了哪些资源消耗呢？），缺点是内存资源消耗严重，因为 MySQL 的内存管理是基于链接的，申请的内存资源只有断开时才会释放。 链接优缺点则相反。

关于长链接的内存资源使用，后面再整理，目前的知识结构认为内存消耗大致有：执行操作将数据页查出放到内存、保存事务的数据视图、维持链接的信息（存疑）

处理:

  1. 定时断开长链接
  2. MySQL >= 5.7 时，可以通过执行 mysql_reset_connetion 初始化链接资源，此操作不需要断开链接但是会将链接恢复到创建完成的状态。

查询缓存

弊大于利的模块，一般不需要开启。 查询缓存将过往的查询语句和结果以 key-value 的形式存储在内存中，当新查询过来时检查是否存在匹配记录，如果命中则在权限校验后直接返回结果。

但是查询缓存有以下致命限制：

  1. 查询缓存的匹配判断是【查询语句完全一致】
  2. 只要对某表执行 insert/update 操作，该表的所有查询缓存将清空 由此导致的结果是查询缓存命中率非常低，往往在频繁查询的静态库时使用。

注意：查询缓存功能在 mysql >= 8.0 被移除。

使用查询缓存

MySQL 也提供“按需使用”的方式。可以将参数 query_cache_type 设置成 DEMAND，由此对于默认的 SQL 语句都不使用查询缓存。而对于确定要使用查询缓存的语句，可以用 SQL_CACHE 显式指定，如下：

select SQL_CACHE * from T where ID=10；

分析器

分析器主要做两件事，词法分析(解析器)和语法分析(预处理器)。 词法分析：解释输入操作中的每个词的含义，生成解析树，比如识别 select 关键字，字段名等； 语法分析：根据词法分析得到的语义解析树，判断操作是否满足 MySQL 语法，比如 SQL 是否书写错误、字段是否存在等。

分析器也会做一次权限验证，叫 precheck。主要针对【能否做某个操作】做校验，比如是否有 create table 权限。

优化器

优化器负责选择索引和确定 join 查询的表连接顺序。

执行器

执行器也会做权限检验，主要针对表级操作，譬如是否有操作某表的权限。 同时执行器会根据表的引擎定义调取对应引擎的接口，搜索引擎的接口诸如“取某个表的第一行”、“取下一行”等，

慢日志的 rows_examined

在 MySQL 慢日志中有个指标 rows_examined，含义就是执行器调用了搜索引擎取数据行的次数
